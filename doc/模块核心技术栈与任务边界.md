# 模块核心技术栈与任务边界（课程项目）

## 1. 文档目的
用于团队快速对齐：
- 每个模块的核心技术栈
- 每个成员应负责的任务边界（做什么 / 不做什么）
- 需要提前补齐的知识点与工具边界

依据文档：
- `.school-doc/知识库智能体开发任务清单.md`
- `.school-doc/知识库智能体项目立项书.md`

说明：结合当前项目口径，运维监控按“两层范围”执行：
- **MVP 必做**：Docker/Compose、健康检查、基础日志与状态排障
- **增强项（时间允许）**：Kubernetes、Prometheus、Grafana

## 3. 各模块边界与知识准备

### 3.1 前端交互模块（G）
- 核心技术栈：`React`、`TypeScript`、路由、状态管理（轻量优先）、HTTP 请求层。
- 任务边界（做）：登录、知识库列表/管理、文档上传、问答页、引用展示、模型切换、会话历史、任务轨迹展示。
- 任务边界（不做）：检索算法、模型推理实现、数据库结构设计。
- 需要掌握：组件拆分、异步请求、错误提示、分页与加载态、基本可用性设计。
- 工具边界：只调用后端公开 API；不直接访问数据库/Qdrant；不实现业务核心规则。
- 交付物：可运行前端页面 + 与后端联调通过的主链路。

### 3.2 后端基础服务模块（L）
- 核心技术栈：`Go`、Web 框架（Gin/Fiber 二选一）、配置管理、日志组件、统一响应结构。
- 任务边界（做）：鉴权基础接口、知识库 CRUD、文档管理接口、会话接口、健康检查接口。
- 任务边界（不做）：前端 UI、复杂运维面板搭建。
- 需要掌握：REST 设计、参数校验、中间件、错误码体系、服务可观测基础。
- 工具边界：对外只暴露稳定 API；内部模块通过 service/repo 分层组织。
- 交付物：稳定 API 服务 + 可供前端直接联调的接口文档。

### 3.3 知识库与文档处理模块（W 主责，L 协同）
- 核心技术栈：`Go` 文档处理链路、文件存储、本地任务处理、文本切片策略。
- 任务边界（做）：上传、解析、清洗、切片、元数据提取、状态跟踪、重建索引。
- 任务边界（不做）：前端展示逻辑、智能体决策流程。
- 需要掌握：PDF/DOCX/TXT/MD 基础解析、切片粒度策略、失败重试和状态机。
- 工具边界：输出统一的 chunk 与 metadata；不负责最终回答生成。
- 交付物：从“上传到可检索”完整流水线。

### 3.4 混合检索模块（L）
- 核心技术栈：`BM25`、`Qdrant`、向量化模型、融合排序策略。
- 任务边界（做）：关键词检索、向量检索、融合去重、引用片段抽取。
- 任务边界（不做）：前端渲染、LLM 最终回答生成。
- 需要掌握：检索召回/精排基础、向量库 schema、命中率与相关性调优。
- 工具边界：对问答/智能体提供统一检索接口；不关心具体页面交互。
- 交付物：可复用检索服务（供问答与智能体共用）。

### 3.5 记忆管理模块（L）
- 核心技术栈：会话摘要策略、短期缓存、长期记忆持久化、召回策略。
- 任务边界（做）：摘要生成、记忆抽取、存储/召回、提问前注入。
- 任务边界（不做）：知识文档切片、前端会话 UI。
- 需要掌握：上下文窗口控制、记忆衰减策略、召回触发条件。
- 工具边界：只输出“可注入上下文”；不直接生成最终答复。
- 交付物：可调试的记忆管理子系统。

### 3.6 智能体编排模块（L）
- 核心技术栈：`ReAct`、Tool Registry、步骤状态记录。
- 任务边界（做）：工具注册、`search_knowledge`/`search_memory`/`summarize_context` 实现、执行轨迹回写。
- 任务边界（不做）：底层数据库运维、前端页面设计。
- 需要掌握：Agent 循环控制、工具调用失败处理、防无限循环策略。
- 工具边界：仅调用已注册工具和模型网关，不直接绕过接口访问底层。
- 交付物：单智能体可运行链路 + 可追踪执行日志。

### 3.7 模型网关模块（L）
- 核心技术栈：外置模型 API、本地模型服务（如 Ollama/vLLM 兼容接口）、统一请求协议。
- 任务边界（做）：多模型接入、参数透传、路由切换、失败降级与耗时统计。
- 任务边界（不做）：具体业务 Prompt 的页面交互。
- 需要掌握：超时重试、限流、成本/时延平衡。
- 工具边界：业务层只能走网关调用模型，禁止模块内各自直连多家模型。
- 交付物：统一模型调用 SDK/服务层。

### 3.8 数据存储模块（L）
- 核心技术栈：MySQL/PostgreSQL（关系数据）、`Qdrant`（向量）、文件系统/对象存储。
- 任务边界（做）：库表设计、索引设计、repo 层封装、初始化脚本。
- 任务边界（不做）：前端页面行为定义。
- 需要掌握：事务一致性、索引优化、向量集合结构与字段规划。
- 工具边界：业务不得跨层直接拼 SQL；统一从 repo/service 入口访问。
- 交付物：稳定可演进的数据层。

### 3.9 部署与运维模块（Z）
- 核心技术栈（MVP）：`Docker`、`Docker Compose`、健康检查、日志收集。
- 核心技术栈（增强）：`Kubernetes`、`Prometheus`、`Grafana`。
- 任务边界（做）：镜像构建、compose 编排、环境变量模板、启动文档、演示环境稳定性。
- 任务边界（不做）：业务功能开发。
- 需要掌握：容器网络、服务依赖编排、日志定位、健康检查策略。
- 工具边界：先保证“可部署+可排障”，监控平台按时间决定是否落地。
- 交付物：一键启动脚本、部署文档、演示运行手册。

### 3.10 测试与文档模块（W 主责，全员协同）
- 核心技术栈：API 测试工具、端到端主链路验证、课程文档写作模板。
- 任务边界（做）：接口测试、检索链路测试、智能体链路测试、部署测试记录、课程文档产出。
- 任务边界（不做）：替代业务模块实现。
- 需要掌握：测试用例设计、回归策略、缺陷追踪、验收材料组织。
- 工具边界：测试应覆盖主链路与关键失败路径；不过度追求复杂自动化框架。
- 交付物：可验收测试记录 + 全套课程过程文档。

## 4. 团队执行建议（防止边界重叠）

1. 所有跨模块能力以“接口契约”对齐，不以口头约定对齐。
2. 每个模块至少产出一份“输入/输出字段说明”。
3. 每周固定一次联调：前端-后端、检索-智能体、部署-测试三条主线。
4. 优先保证 MVP 主链路可演示，再做增强项（K8s/Prometheus/Grafana）。

## 5. 个人开工前最小准备清单

- G：先跑通前端脚手架 + 2 个页面 mock 联调。
- L：先定义统一 API 响应与错误码，再并行推进检索/智能体。
- W：先确定文档解析与切片格式，再写测试样例。
- Z：先交付本地一键部署脚本，再评估监控增强是否可加。
